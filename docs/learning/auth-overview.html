3. OIDC とは

OIDC (OpenID Connect) は、「自分のアプリで認証処理をやらず、外部 (Google) に代行してもらう」仕組み。 パスワード管理を自前でやらなくて済む。

用語

用語意味nonda の場合
IdP認証を担当する外部サービスGoogle
RP (Relying Party)IdP を利用するアプリnonda (Web)
Client IDGoogle が RP に発行する公開 IDAUTH_GOOGLE_ID
Client SecretGoogle が RP に発行する秘密鍵AUTH_GOOGLE_SECRET
Redirect URIGoogle が認可後に戻す URL (事前登録)/api/auth/callback/google
Scope取得したい情報の種類openid email profile
Authorization CodeGoogle が一時的に発行する引換券URL の ?code=...
ID Tokenユーザー情報入りの JWTemail, name, picture, sub を含む
subIdP 側のユーザー一意 IDGoogle アカウントの ID

処理フロー

nonda 側の準備（事前作業）

  1. GCP で OAuth クライアントを作成 → AUTH_GOOGLE_ID と AUTH_GOOGLE_SECRET を取得
  2. 承認済みリダイレクト URI に {オリジン}/api/auth/callback/google を登録
  3. 取得した ID / Secret を Doppler に保存

なぜ Authorization Code を経由するか

  • id_token を直接ブラウザに渡すと、URL から盗まれる可能性がある
  • code はブラウザを経由するが、id_token と引き換えるには client_secret も必要 (Web サーバーしか持っていない)
  • code が漏れても client_secret なしでは id_token を取れない → 安全

nonda での実装

  • ③〜⑧ の処理は Auth.js が全部やってくれる
  • アプリ側のコードは「signIn("google") を呼ぶ」「auth() でセッション取得」程度
  • 取得した id_token の sub や email をもとに、users / accounts テーブルに保存（DrizzleAdapter が担当）

4. ログイン処理の流れ

1
ブラウザ /login で「Google でログイン」ボタンを押す
2
Web → Google Auth.js が Google ログイン画面へリダイレクト
3
Google ユーザーが承認 → Web の /api/auth/callback/google に戻す
4
Web → DB users / accounts レコードを作成（初回ログインのみ）
5
Web → ブラウザ セッショントークンを Cookie に保存
6
ブラウザ / へリダイレクト → ホーム画面表示

5. API を呼ぶ流れ

1
ブラウザ ユーザーがページを開く
2
Web Cookie から userId を取得（auth() 関数）
3
Web JWT を発行（sub: userId, 5 分有効, AUTH_SECRET で署名）
4
Web → API Authorization: Bearer <jwt> を付けて fetch
5
API ヘッダから JWT を取り出して AUTH_SECRET で署名を検証
6
API → DB payload.sub から userId を取り、SQL 実行
7
API → Web → ブラウザ 結果を返す

6. なぜ Cookie ではなく JWT を発行するか

  • Web と API は別ドメインなので、Cookie は API には自動送信されない
  • API 側は短い文字列を検証するだけで済む
  • JWT を短命 (5 分) にすることで、漏洩時のリスクを抑えられる

7. JWT の構造

Header
署名アルゴリズム
{"alg":"HS256","typ":"JWT"}
Payload
データ
{"sub":"uuid","exp":12345}
Signature
署名
HMAC-SHA256(Header + Payload, AUTH_SECRET)
↓ . でつないで 1 つの文字列
eyJhbGc....eyJzdWI....a1b2c3...

AUTH_SECRET を知らない人は正しい Signature を作れない。改ざんすると検証で弾かれる。

12. よくある誤解

  • Auth.js の Cookie をそのまま API に送ればよい
    → 別ドメインなので Cookie は API に届かない。Web で JWT に変換する。
  • API も Auth.js を使えばよい
    → Auth.js は Next.js 向け。API では JWT 検証だけ自前で行う。
  • JWT は暗号化されている
    → HS256 は署名のみ。Payload は base64 デコードすれば誰でも読める。秘密情報は入れない。
